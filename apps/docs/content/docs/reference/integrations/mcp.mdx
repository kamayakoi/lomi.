# MCP integration

Use Lomi MCP to let tools like Cursor and Claude call your merchant API through a standardized MCP interface.

## What this server provides

- OpenAPI-driven MCP tools generated from Lomi public API
- Hosted HTTP streamable MCP endpoint
- Local stdio mode for development
- Optional MCP bearer gate (`LOMI_MCP_BEARER_TOKEN`)

## Hosted endpoint for your users

For real users, deploy one hosted MCP service (for example on Railway) and share the MCP URL.

Default endpoint:

- MCP: `https://<your-host>/mcp`
- Health: `https://<your-host>/health`

## Authentication model

### Merchant credential (REST)

Tool calls need a valid merchant API key (same keys as the REST API):

1. **Recommended:** `x-lomi-api-key` or `x-api-key` on MCP HTTP requests (especially when `LOMI_MCP_BEARER_TOKEN` protects the route).
2. **Optional:** `Authorization: Bearer <lomi_* key>` is accepted **only when** `LOMI_MCP_BEARER_TOKEN` is **not** configured (typical local/dev setups).

### MCP transport gate

When set, clients must send:

- `Authorization: Bearer <LOMI_MCP_BEARER_TOKEN>`

`/health` stays unauthenticated for probes.

<Callout type="info">
  For multi-merchant production, avoid a single server-wide `LOMI_API_KEY`; pass per-user keys via headers instead.
</Callout>

## Dashboard connect flow (device authorization)

Merchants can approve MCP access in the browser—similar to CLI login—without hunting for an existing API key copy/paste:

1. **Start session:** `POST https://<supabase-project>.supabase.co/functions/v1/mcp-auth/device-auth`  
   Response includes `device_code`, `user_code`, `verification_uri`, `interval`, `expires_in`.
2. **Browser:** User opens `verification_uri` (`/connect/mcp-verify` on the dashboard), signs in if needed, enters `user_code`, and authorizes.
3. **Poll token:** `POST …/mcp-auth/token` with JSON `{ "device_code": "<device_code>" }` until it returns `{ "api_key": "lomi_mcp_…", "token_type": "bearer" }` (while pending, the endpoint responds like OAuth device flow with `authorization_pending`).
4. **Use with MCP:** Configure the HTTP MCP client to send `x-lomi-api-key: <api_key>` on initialize (and transport bearer if your host requires it).

**Deploy requirements:** apply the `mcp_device_requests` migration and deploy the **`mcp-auth`** Supabase Edge Function (`apps/dashboard/supabase/functions/mcp-auth`).

## Railway deployment (quick path)

1. Create a new Railway service from this repository.
2. Set root directory to `apps/mcp`.
3. Deploy with start command `pnpm run start:http` (already configured in `apps/mcp/railway.json`).
4. Configure env vars:
   - required: `LOMI_MCP_BEARER_TOKEN`
   - recommended: `LOMI_API_BASE_URL`, `LOMI_API_BASE_URL_ALLOWLIST`
   - optional fallback: `LOMI_API_KEY`
5. Confirm `GET /health` is `200`.

## Client configuration expectations

Hosted MCP users typically configure:

- MCP URL (`https://<host>/mcp` or custom `LOMI_MCP_HTTP_PATH`)
- Optional transport bearer (`LOMI_MCP_BEARER_TOKEN`)
- Merchant credential header (`x-lomi-api-key`) from the dashboard connect flow or from **Developers → API keys**

See `apps/mcp/README.md` for Cursor / HTTP examples.
