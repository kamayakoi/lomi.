title: Payment state machine
description: How transaction and provider statuses transition, expire, and map to balances.

This page is the canonical behavior reference for payment lifecycle logic in lomi. It aligns with server-side functions in `create_transaction`, `update_transaction_status`, expiration jobs, and balance updates.

## Transaction statuses

Common `transaction_status` values you will see:

| Status | Meaning |
| --- | --- |
| `pending` | Payment initiated; final outcome not yet confirmed (typical in **live**). |
| `completed` | Payment succeeded; merchant balance and fees are settled per platform rules. |
| `failed` | Payment did not succeed (including some expiration paths). |
| `expired` | Pending payment timed out (alternative to `failed`, depending on expiration job used). |
| `refunded` | Refund processed against the original payment. |

### Test vs live

When a transaction is created:

- In **test** environment, status may be **`completed` immediately** (no asynchronous provider confirmation).
- In **live** environment, new payments typically start **`pending`** until the provider confirms success.

## Status updates: idempotency and metadata

When status moves to `completed`:

- **Duplicate completion is ignored** for the balance path: if completion is attempted again, the row is still updated with **merged** metadata (new provider IDs are preserved), but balance is not double-applied.
- **Metadata is merged**, not replaced, so webhook retries can append fields safely.

## Provider-facing status mapping

Platform `transaction_status` is reflected in `provider_payment_status` on linked provider records, for example:

| Transaction status | Provider payment status |
| --- | --- |
| `completed` | `succeeded` |
| `pending` | `processing` |
| `failed` | `cancelled` |
| `expired` | `expired` |
| `refunded` | `refunded` |

Exact field names in API responses follow your OpenAPI / resource models; see [Transactions](/reference/payments/transactions).

## Pending expiration

Pending transactions can be expired by scheduled jobs:

- By **age** (e.g. older than N hours).
- Optionally also by **provider session expiry** (e.g. Wave session `when_expires`), depending on which expiration function is used.

Expired rows carry `expiration_info` in metadata describing why they closed.

## What happens on `completed`

In order:

1. **Coupon usage** may be incremented for completed transactions.
2. **Balances** are updated via a single idempotent path that checks `metadata.balance_updated` to avoid double crediting.
3. **Subscriptions** linked to the transaction may move from `pending` to `active` when applicable.

For balance and availability timing, see [Balance and settlement](/reference/platform/balance-and-settlement).
