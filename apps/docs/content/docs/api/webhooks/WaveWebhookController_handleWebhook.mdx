## Overview

**Inbound Wave→lomi webhook receiver.** This is called by Wave’s servers with a JSON body; lomi verifies signatures using **`Wave-Signature`** (v1 scheme) or falls back to legacy `Authorization` headers. **Not** a merchant `X-API-KEY` route.

See verification flow in `apps/api/src/webhooks/providers/wave/wave-webhook.service.ts` (looks for headers containing `wave-signature`, etc.).

## Authentication

Wave signs requests — present **`Wave-Signature`** (preferred) or `Wave-Authorization`/`Authorization` per legacy behavior. Merchant API keys **must not** be substituted here.

## Endpoint

`POST /webhooks/wave`

Base URLs:

- `https://sandbox.api.lomi.africa`
- `https://api.lomi.africa`

## Request

### Path parameters

_No path parameters._

### Query parameters

_None._

### Request body

Wave event JSON. The controller prefers `rawBody` (Buffer) parsed to JSON; fallback uses parsed `body` if raw is absent.

## Responses

| Status | Description |
| --- | --- |
| `200` | Webhook processed — `{ received: true, … }` |
| `401` | Signature verification failed (`UnauthorizedException`) |
| `500` | Processing error |

## Errors

**401** when signature cannot be validated (deployment should configure Wave webhook secret correctly).

## Example

Operational teams exercise via Wave dashboards or tooling that preserves signing headers—not raw merchant curl.

```bash
curl -sS -X POST "https://sandbox.api.lomi.africa/webhooks/wave" \
  -H "Content-Type: application/json" \
  -H "Wave-Signature: t=1234567890,v1=hexsignature" \
  --data-binary @wave-event.json
```
